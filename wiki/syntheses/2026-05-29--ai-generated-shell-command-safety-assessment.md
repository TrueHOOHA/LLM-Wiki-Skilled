# AI-generated Shell Command Safety Assessment

## Question / Purpose
Assess [[Nick Radford tweet on ffmgen AI FFmpeg command generator]] as a weak social source and preserve the reusable [[AI Command-generation Safety Pattern]] without recommending adoption, implementation, prototype work, or direct Hermes workflow changes.

## Answer / Analysis
Strongest counterargument first: this source proves very little about the correctness or safety of AI-generated FFmpeg commands. A visible app, endpoint, model metadata, warning text, and client-side filters establish that [[ffmgen]] exists and has some safety affordances, but they do not show that commands are correct, safely quoted, non-destructive, portable, robust against prompt injection, or safe to execute on real media files.

The useful Tolaria payload is the guardrail pattern. Natural-language-to-command tools sit at a dangerous boundary: a plausible answer can overwrite files, recursively process the wrong paths, leak data, run a network transfer, or embed unsafe shell expansion. For FFmpeg specifically, the common tasks are attractive because they are concrete and repetitive, but correctness still depends on codecs, container compatibility, file paths, filter graphs, metadata, overwrite behavior, and local FFmpeg version.

The captured ffmgen warning is directionally correct: "Commands are generated by AI. Always review before running." It is insufficient by itself. A stronger command-generation interface should pair the generated command with plain-language explanation, risk labels, expected inputs/outputs, safe quoting, overwrite/destructive-operation warnings, dry-run or probe guidance where possible, and test fixtures or local verification steps. Prompt-injection filtering should be server-side and evidence-backed; client-side phrase blocks and prompt-length limits are useful UX signals but weak safety boundaries. The observed reasoning-stream exposure is also a safety/UX concern because command tools should avoid leaking hidden model process or prompt details unless deliberately designed and reviewed.

## Implications
- Preserve ffmgen as a weak/social example of AI command generation, not as a recommended tool.
- Treat generated shell commands as executable-risk artifacts: review, explain, quote, verify, and gate them before execution.
- Do not rely on client-side prompt filters as the safety boundary; they are visible affordances only.
- Do not prototype, install, integrate, or route Hermes workflow changes from this card. Any practical evaluation of NL-to-shell tools should be separately approved by Overseer.
