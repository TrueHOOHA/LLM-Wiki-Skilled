## Primary article text captured from X DOM (abridged but source-faithful)

The article states that Markdown has become the dominant file format agents use to communicate with humans because it is simple, portable, rich enough for text, and easy to edit. Thariq argues that as agents become more powerful, Markdown becomes restrictive: long Markdown files are hard to read, richer visualizations/color/diagrams are needed, and he increasingly uses generated files as specs, reference files, brainstorming outputs, and plans rather than editing them directly.

Core claim: he has started preferring HTML as an output format instead of Markdown, and sees this increasingly among others on the Claude Code team.

Reasons given:

- Information density: HTML can express document structure, tabular data, CSS design data, SVG illustrations, code snippets, interactions via JS/CSS, workflow diagrams, spatial data via absolute positioning/canvas, and images. He claims almost anything Claude can read can be represented fairly efficiently with HTML.
- Visual clarity and ease of reading: Claude writes larger specs/plans as work gets complex; he finds he does not read 100+ line Markdown files carefully, and colleagues are less likely to read them. HTML documents can use visual structure, tabs, illustrations, links, and responsive layout.
- Ease of sharing: Markdown is harder to share because browsers do not render it natively; uploaded HTML files can be shared as links.
- Two-way interaction: HTML can include sliders, knobs, controls, copy buttons, and other interfaces that let the human adjust choices and paste the result back into Claude Code.
- Data ingestion: Claude Code can read the filesystem, MCP context such as Slack/Linear, browser context, git history, etc., then generate context-rich HTML reports/explainers.
- Joy: he says HTML artifacts are more fun and keep him more invested in the creation.

He cautions that people may try to turn this into a `/html` skill, but says the minimal starting point is simply asking Claude Code to "make a HTML file" or "make a HTML artifact" and learning which artifact should do what.

Use cases listed:

- Specs, planning, and exploration: ask Claude Code to generate multiple approaches or a thorough implementation plan as an HTML file with mockups, data flow, and code snippets. Pass generated files into a new session or verification agent later.
- Code review and understanding: render diffs, annotations, flowcharts, modules, severity color-coding, and PR explainers; he says he attaches an HTML code explainer to every PR.
- Design and prototypes: use HTML to sketch designs, prototype interactions, tune animations with sliders/knobs, and copy selected parameters.
- Reports, research, and learning: synthesize codebase/git/Slack/internet context into readable reports, explainers, diagrams, decks, status reports, incident reports, and leadership/team documents.
- Custom editing interfaces: purpose-built single HTML files for reordering tickets, editing feature flags/configs, tuning prompts/templates, curating datasets, annotating documents/diffs, or choosing hard-to-express values. The key pattern is to end with export/copy-as-JSON/copy-as-prompt/copy-diff output back to Claude Code.

FAQ points captured:

- Token efficiency: HTML may use more tokens, but he argues expressiveness and higher chance of human reading are worth it; he references a 1MM context window in Opus 4.7.
- Markdown: he says he has mostly stopped using Markdown, but recognizes he may be on the maximalist side.
- Viewing: open locally in a browser or upload to S3 for sharing.
- Generation time: HTML may take 2-4x longer than Markdown.
- Version control: HTML diffs are noisy and harder to review.
- Taste/design: a frontend design plugin or a single design-system HTML reference can help Claude match style.

## Source assessment

- What the source claims: HTML can be a better output medium than Markdown for many Claude Code workflows because it supports dense visual structure, interactivity, sharing, and two-way human-agent feedback.
- Evidence actually provided: practitioner essay, Claude Blog mirror, a public repo/gallery with 20 self-contained example artifacts, screenshots/media, and a related playground article. No benchmark or controlled comparison.
- Concepts worth independent investigation: HTML artifacts as agent handoff/review surfaces; interactive micro-UIs that export prompts/diffs/JSON back to the agent; artifact packs for implementation planning, PR review, incident reports, and Tolaria syntheses; safety/review concerns for generated JavaScript; version-control friction for generated HTML.
- Recommended Alpha action: archive and queue one Beta Tolaria knowledge task to synthesize the HTML-artifact workflow pattern and propose how/when it should affect Tolaria/Hermes deliverable conventions. Do not queue implementation or skill/config changes without Overseer approval.

## Contradiction notes / caveats

- Social source and personal essay are persuasive but not empirical; evidence grade remains weak-to-medium depending on whether the repo/gallery is treated as primary demonstration.
- HTML output can be token-heavier, slower to generate, noisier in version control, and riskier if generated JavaScript is shared/executed without review.
- The post itself warns against prematurely turning this into a generic `/html` skill; the useful part is deciding artifact purpose and export loop, not blindly converting every Markdown output to HTML.
